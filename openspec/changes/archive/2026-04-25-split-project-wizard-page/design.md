## Context

現在の `frontend/src/pages/project-wizard-page.tsx` は、ウィザード全体の状態管理、API呼び出し、ステップ進行、プリセット選択、必須項目フォーム、任意項目フォーム、確認画面、ナビゲーションボタンを1ファイルに含んでいる。MVPの動作は成立しているが、項目追加やUI調整のたびにページ全体を読む必要があり、回帰リスクが高い。

この変更はフロントエンド内部の分割であり、バックエンドAPI、URL、4ステップ構成、表示文言、生成後の遷移は維持する。PC専用・日本語のみというMVP前提も変えない。

## Goals / Non-Goals

**Goals:**

- `ProjectWizardPage` をルートコンテナとして残し、API連携と画面組み立てに集中させる
- ステップ進行表示、プリセット選択、必須項目、任意項目、確認画面を小さなコンポーネントに分ける
- 初期回答、必須項目、選択肢、ステップ定義を定数として切り出す
- `AnswerMap` 更新ロジックやステップバリデーションをテストしやすいヘルパーに整理する
- 既存E2Eで守られているプリセット反映、生成、編集後ZIPダウンロードの流れを維持する

**Non-Goals:**

- ウィザードのステップ数や入力項目の追加・削除
- 表示文言や視覚デザインの大幅変更
- バックエンドAPI、DB、生成処理の変更
- ルーティング構成や認証フローの変更

## Decisions

1. `project-wizard-page.tsx` はエクスポート名とルート責務を維持する。
   - 理由: 既存ルーティングやテストの参照先を変えずに、分割の影響を局所化できる。
   - 代替案: ページ自体をフォルダ内の `index.tsx` に移す。今回は import 経路変更の必要がなく、既存構成に合わせて最小差分にする。

2. ページ専用コンポーネントは `frontend/src/pages/project-wizard/` 配下に置く。
   - 理由: 汎用UIではなくウィザード固有の部品であり、`components/ui` や共通 `components` に置くと再利用範囲が曖昧になる。
   - 代替案: `frontend/src/components/project-wizard/` に置く。将来ほかのページで再利用する段階までは、ページ配下に閉じる方が責務が明確。

3. 状態は現状通りページコンテナで管理し、子コンポーネントは値とコールバックを受け取る。
   - 理由: React Contextや状態管理ライブラリを追加せず、API呼び出し順序と生成フローを保ちやすい。
   - 代替案: カスタムhookへ全状態を移す。分割後にページがまだ大きい場合の追加整理として扱う。

4. バリデーションと回答更新の小さなヘルパーだけを純粋関数化する。
   - 理由: 必須項目判定や配列トグルは回帰しやすい一方、API呼び出しは既存E2Eで十分に確認できる。
   - 代替案: フォームライブラリを導入する。新規依存が増え、MVPの単純な入力構造には過剰。

## Risks / Trade-offs

- [propsの受け渡しが多くなる] → ステップ単位でpropsをまとめ、共通型を定義して読みやすさを保つ
- [表示文言やアクセシブルなラベルが変わる] → 既存E2Eが参照するラベル・ボタン名・プリセット名を維持する
- [プリセット適用時の初期化挙動が変わる] → `initialAnswers` と `preset.answers` のマージ順を維持する
- [生成フローのAPI呼び出し順が変わる] → `ensureProject`、`updateAnswers`、`generateFiles`、詳細ページ遷移の順序を維持する

## Migration Plan

データ移行は不要。既存ページを段階的に分割し、E2Eテストと型チェックを通してから通常デプロイする。問題が出た場合はフロントエンドの分割差分のみを戻せば、保存済みデータやバックエンドへの復旧作業は不要。
