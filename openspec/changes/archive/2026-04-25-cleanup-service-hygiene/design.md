## Context

Harness Builder のバックエンドは、ウィザード回答を保存し、Jinja2テンプレートからハーネスファイルを生成し、保存済みファイルをZIPとしてエクスポートする。現在のサービス層は小規模で動作している一方、`generator.py` にテンプレート選択、コンテキスト作成、既存ファイル更新、レンダリングがまとまっており、今後テンプレートや出力形式を増やすと変更時の見通しが悪くなる。

この変更は内部整理であり、既存のFastAPIエンドポイント、レスポンス形式、DBスキーマ、生成ファイルパス、テンプレート内容は維持する。MVPの3秒以内レスポンス要件と日本語のみの仕様にも影響させない。

## Goals / Non-Goals

**Goals:**

- `answers.py`, `generator.py`, `export.py` の責務を読みやすい単位に分ける
- テンプレートコンテキスト作成、テンプレート選択、既存生成ファイル更新、ZIP書き込みをテストしやすくする
- 既存の生成・編集・エクスポートフローの互換性を回帰テストで守る
- 新規外部依存を追加せず、既存の同期SQLAlchemyセッション利用を維持する

**Non-Goals:**

- APIパス、レスポンススキーマ、DBスキーマの変更
- テンプレート本文や生成ファイルの種類の変更
- フロントエンドUIの変更
- AIツール選択を変更した再生成時に、過去に生成済みの非選択ツール用ファイルを削除する挙動の追加

## Decisions

1. サービス関数ベースの構成を維持する。
   - 理由: 現在のルート層は関数型サービスを直接呼び出しており、クラス化やDIコンテナ導入は変更範囲に対して重い。
   - 代替案: サービスクラスへ全面移行する。今回は公開挙動に関係しない構造変更が大きくなるため採用しない。

2. `generator.py` は小さな純粋関数とDB更新関数に分ける。
   - 理由: 回答値のリスト化、テンプレート定義の選択、テンプレート描画、既存ファイルの更新判断を分離すると、テスト対象が明確になる。
   - 代替案: 既存の `generate_project_files` 内にコメントを追加して整理する。処理の依存関係は残るため、今後のテンプレート追加時の安全性が上がりにくい。

3. Jinja2環境とテンプレート定義は既存の仕組みを維持する。
   - 理由: 生成内容の互換性が最優先であり、テンプレート検索パスやautoescape設定を変える必要はない。
   - 代替案: グローバルキャッシュや別テンプレートローダーを導入する。性能上の必要がまだなく、差分のリスクが上がる。

4. ZIP作成は保存済み `GeneratedFile` の `file_path` と `content` をそのまま書き込む。
   - 理由: 既存のエクスポート要件は保存済み生成ファイルの構造と編集後内容を保持することにある。
   - 代替案: パス検証や正規化を追加する。セキュリティ上の別課題として扱うべきで、今回は互換性を優先する。

## Risks / Trade-offs

- [生成内容の微小差分] → 既存テストに加えて、代表的な生成ファイルパスと再生成時の重複防止を検証する
- [暗黙の順序依存] → APIで保証している一覧順とZIP内容の検証を分け、必要以上に出力順を変更しない
- [内部整理に見せかけた仕様変更] → 非選択ツールの過去生成ファイル削除など、挙動変更になる処理は今回含めない
- [テストが実装詳細に寄りすぎる] → 公開APIのE2E寄りテストと、純粋関数の小さな単体テストのバランスを取る

## Migration Plan

DBスキーマ変更とデータ移行は不要。既存テストを通したうえで通常デプロイできる。問題が出た場合はサービス層の整理差分のみを戻せば、保存済みデータへの復旧作業は不要。
