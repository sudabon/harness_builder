## Context

Harness Builder のウィザード回答は、`project_kind`, `languages`, `frameworks`, `ai_tools`, `test_strategy`, `lint_format`, `prohibited_actions`, `review_policy`, `branch_strategy`, `ci_command`, `deploy_constraints`, `security_requirements`, `failure_examples`, `naming_convention` というキーを中心に構成されている。現在はフロントエンドの `project-wizard-page.tsx`、バックエンドの `projects.py`、`presets.py`/`presets.py` 相当の定義、生成サービス、テンプレートが同じキーを個別に参照している。

この変更はスキーマ一元化であり、API payload、DB保存形式、生成テンプレートが受け取るキー名は変更しない。ユーザーから見るウィザード項目とプリセットの挙動も維持する。

## Goals / Non-Goals

**Goals:**

- 質問キー、型、必須/任意、選択肢、初期値、表示ラベルを正規スキーマとして定義する
- フロントエンドの初期値、選択肢、ステップ表示、必須バリデーションをスキーマ由来にする
- バックエンドの生成前必須チェックとプリセット整合性チェックをスキーマ由来にする
- プリセットが正規スキーマに存在するキーと許可された値だけを使っていることをテストで検知する
- 既存の回答保存・取得・生成フローを維持する

**Non-Goals:**

- API payloadを配列形式やネスト形式に変更すること
- `questionnaire_answers` テーブル構造の変更
- 既存フィールド名のリネーム
- ウィザード項目やプリセット内容の追加・削除
- フロントエンドとバックエンドのコード生成パイプライン導入

## Decisions

1. フロントエンドとバックエンドにそれぞれ言語ネイティブなスキーマ定義を置き、テストで整合性を守る。
   - 理由: TypeScriptとPythonの両方で型安全に扱いやすく、ビルド手順やコード生成依存を増やさずに済む。
   - 代替案: JSONファイルを共有して両方から読み込む。単一ソースにはなるが、型補完や実行時パス管理が複雑になる。

2. バックエンドは既存の `answers: dict[str, Any]` 形式を維持しつつ、既知キーの型・必須性・選択肢を検証する。
   - 理由: API互換性を保ちながら、生成に必要な入力の不整合を早期に検知できる。
   - 代替案: Pydanticモデルをフィールド固定で公開する。型は明確になるが、既存の柔軟なJSON payloadからの移行が大きい。

3. 未知キーの扱いは破壊的変更にしない。
   - 理由: 現在のAPIは任意キーを保存できるため、即座に拒否すると互換性リスクがある。
   - 代替案: 未知キーを400で拒否する。将来的には検討できるが、この変更では既存保存形式を維持する。

4. 生成サービスはスキーマ定義から配列型・文字列型を判断して正規化する。
   - 理由: `build_template_context` の `_as_list` と個別キー参照をスキーマに寄せることで、項目追加時の変更箇所を減らせる。
   - 代替案: 生成サービスは現状維持し、ウィザードだけを整理する。重複が残り、変更目的を満たしにくい。

## Risks / Trade-offs

- [二重スキーマの不整合] → フロントエンド/バックエンドのキー、型、必須性、選択肢を比較するテストを追加する
- [型検証で既存payloadを壊す] → 既存キー・既存値・既存保存形式を優先し、未知キーは破壊的に拒否しない
- [プリセットが必須自由記述を埋めていない] → 既存プリセットの挙動を維持し、プリセットは全必須項目ではなく事前入力対象の必須選択項目との整合性を検証する
- [テンプレート入力名の変更漏れ] → 生成テストで代表テンプレート出力と既存ファイル生成フローを確認する

## Migration Plan

1. 現在の質問キー、型、必須/任意、選択肢を棚卸しする。
2. バックエンドに正規質問スキーマとヘルパーを追加する。
3. フロントエンドに対応する質問スキーマと型を追加し、既存定数を置き換える。
4. プリセット、必須チェック、生成コンテキストをスキーマ参照へ寄せる。
5. スキーマ整合性、プリセット整合性、既存ウィザード/生成フローのテストを実行する。

DBマイグレーションは不要。問題が出た場合はスキーマ参照への置き換え差分を戻せば、保存済みデータへの復旧作業は不要。
